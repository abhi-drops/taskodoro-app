# Backend + Sync Implementation Plan

## Overview

Add an optional cloud sync feature to the existing Capacitor + React app.
- **Offline-first**: app works fully without internet, local state is always source of truth
- **Optional sync**: user enables sync, signs in with Google, data is linked to Neon DB
- **Cross-platform**: same Vite build works on Android (Capacitor) and web browser
- **Future-ready**: Chrome extension can plug into the same sync API

### Key design decisions
- Every row has `userId` — row-level security enforced in all queries

## Sync Model: Last-Write-Wins per Entity

### Why not operation log (git-like)?

An operation log (storing every action) has issues:
- Log grows forever, gets expensive
- Clock skew between devices causes wrong ordering
- Replaying 1000 operations is complex

**Better approach for a todo app**: store current state per entity, track `updatedAt` per row.
Server timestamp = authoritative. Client timestamp = ignored for ordering.

You can still add an audit log table later if you want history — but it won't drive sync.

### Sync flow

```
Client                          Server
  |                               |
  |-- POST /api/sync ------------>|
  |   { lastSyncAt, localState }  |
  |                               |-- validate JWT
  |                               |-- for each entity in localState:
  |                               |     if server row doesn't exist → insert
  |                               |     if server updatedAt > client updatedAt → server wins
  |                               |     if client updatedAt > server updatedAt → client wins, update row
  |                               |-- return full server state + new syncTimestamp
  |<-- { serverState, syncAt } ---|
  |                               |
  |-- merge serverState into      |
  |   local store                 |
  |-- save syncAt to storage      |
```

### First sync (new user)

1. User signs in with Google
2. Server has no data for this user
3. Client sends full local state
4. Server inserts everything, returns it back with server timestamps
5. Client saves `syncAt`

### Subsequent syncs

1. Client sends `lastSyncAt` + current local state
2. Server compares `updatedAt` per entity
3. Returns merged state

### Conflict resolution

Simple: **higher `updatedAt` wins per entity** (server-assigned).

If device A edits a todo offline and device B also edits it online:
- Device B's edit already reached the server → has a server `updatedAt`
- Device A syncs → server compares timestamps → whichever was more recent wins

This is acceptable for a personal todo app. No complex CRDT needed.

## Auth Flow

### Web (browser)
1. User clicks "Sign in with Google"
2. Better Auth redirects to Google OAuth
3. Google redirects back to `/auth/callback`
4. Better Auth creates session, sets cookie + returns JWT
5. Client stores JWT in localStorage

### Android (Capacitor)
1. User clicks "Sign in with Google"
2. `@codetrix-studio/capacitor-google-auth` opens native Google sign-in sheet
3. Returns `idToken` (Google JWT)
4. Client sends `idToken` to `POST /auth/google/native`
5. Server verifies token with Google, creates Better Auth session, returns app JWT
6. Client stores JWT in `@capacitor/preferences`

## Live Timer Sync (Cross-Platform)

### The problem

Todo sync can be lazy (eventual consistency is fine). Timer sync cannot — if the phone shows 12:34 remaining and the web tab shows 12:31, that's broken UX. It must be real-time and drift-free.

### Solution: broadcast start time, compute locally

Never transmit a countdown. Transmit **when the block started** and let every client compute remaining time independently using math.

```
remaining = blockDurationMs - (accumuledMs + (now - startedAt))
```

All clients stay in sync automatically because they all use the same `startedAt`. No drift possible.

### Timer state (one row per user in DB)

```
timer_state — userId, status ('running'|'paused'|'idle'), blockIndex,
              startedAt (epoch ms, server-assigned), pausedAt (epoch ms | null),
              accumuledMs (ms elapsed before last resume), blockDurationMs
```

### Real-time transport: SSE (Server-Sent Events)

Each client holds an open SSE connection to `GET /api/timer/stream`.
When any device performs a timer action, the server broadcasts the new timer state to all connected clients for that user.

```
User pauses timer on phone
  └── POST /api/timer/pause
        └── server updates timer_state row
        └── server broadcasts SSE event to all this user's open connections
              └── web tab receives → recomputes UI instantly
              └── extension receives → updates badge/popup
              └── other devices receive → update UI
```

**Why SSE over WebSocket**: Timer state only flows server → client. SSE is simpler, has automatic browser reconnection, and Hono supports it natively. No need for bidirectional streaming.

### SSE event shape

```json
{
  "type": "timer_update",
  "status": "running",
  "blockIndex": 2,
  "startedAt": 1713600000000,
  "pausedAt": null,
  "accumuledMs": 0,
  "blockDurationMs": 1500000
}
```

### Timer API endpoints

```
POST /api/timer/start    — start or restart a block
POST /api/timer/pause    — pause current block
POST /api/timer/resume   — resume paused block
POST /api/timer/next     — skip to next block
POST /api/timer/stop     — end session
GET  /api/timer/state    — fetch current state (used on foreground re-entry)
GET  /api/timer/stream   — SSE connection (persistent, per client)
```

### Android background timer

Your existing `PomodoroTimerService` (native foreground service) still runs in background — it's more reliable than a web connection for keeping the OS alive. The relationship is:

- **Background**: native service is source of truth, ticks independently
- **Foreground re-entry**: app calls `GET /api/timer/state` once to resync, then reopens SSE connection
- **User action (pause/skip)**: both native service and server are updated together

SSE connections are dropped when Android backgrounds the app — that's expected and fine.

### Conflict resolution

Last write wins — server timestamp decides. If two devices somehow both send `start` simultaneously, whichever request arrives at the server last sets the `startedAt`, and SSE broadcasts the final state to both.

### New files for timer sync

```
server/src/routes/timer.ts      # all timer endpoints + SSE stream
server/src/lib/timerBroadcast.ts # in-memory SSE connection registry per userId
src/hooks/useTimerSync.ts        # client: opens SSE, applies server state to local timer
```

## Key Decisions Summary

1. **Monorepo** — backend in `server/`, same git repo, simpler to manage
2. **Server-assigned timestamps** — avoids clock skew between devices
3. **Last-write-wins per entity** — simple, correct, good enough for personal todo app
4. **Better Auth self-hosted** — free, no vendor lock-in, data stays in your Neon DB
5. **Same Vite build** — works on Android (Capacitor) and web via platform adapter
6. **Offline-first** — local store always works, sync is additive, never blocking
7. **Timer sync via SSE** — broadcast `startedAt`, clients compute remaining locally — zero drift, no countdown transmitted
